Features

  • Native 32 bit Windows application: long filenames supported, editable file size limited only by available system memory.
  • Small program size, loads fast.
  • Cut, copy & paste binary values. Syntax for describing byte, word, longword, float and double binary values in text form to use with find and paste functions.
  • Directly enter hex values or text in main window (overwrite or insert).
  • Compare files.
  • Search for any combination of text and binary values.
  • Bit manipulation.
  • Import/Export as hexdump to/from file or clipboard.
  • Automatically adjust bytes displayed per hexdump line to window width, or set bytes per line manually.
  • Choose font size, character set (ANSI or OEM), colors of text and background.
  • Intel or Motorola binary format.
  • Can be used as "Send to"-Target and in the context menu for all files and for files with non-registered extensions.
  • Set bookmarks to easily access offsets in the file.
  • Load files partially.
  • Load files by dragging them into the window.
  • Apply binary templates for structured information.
  • Easily remove links to frhed, remove frhed's registry entries and upgrade from frhed versions 1.0.154 and upward (1.0.153 and below used ini files instead) from within the program itself.
  • Switch between sets of instance options.
  • Automatically create backups when saving.
  • OLE drag-drop
  • Source code is included. (Written with Microsoft Visual C++. Does not use MFC, only Win32 API calls and C library calls, uses C++ classes).
  • No registration fee required!

Binary Files

  • What are binary files?
  • What are bits and bytes?

What are binary files?

All the files on your computer, programs as well as documents, are really large "lists" of numbers. Imagine a numbered list like this one:

0) 24
1) 33
2) 9
3) 0
4) 115

The numbering starts at zero. At each list index there is a number, for example the number 9 is at index "2". The index is usually called "offset". frhed displays the current offset in the lower left corner of it's window. Large files contain many numbers, small files not so many. The numbers in the list can have a value between 0 and 255, for a special reason: Each number is a "byte".

What are bits and bytes?

"Bit" is an abbreviation for "BInary digiT". It can have a value of either 0 or 1. One bit alone is not very useful, because already the number 2 can not be described with it. So you take many bits to describe large numbers. This is how it works:
The normal decimal system you know, with digits from 0 to 9, describes numbers larger than 9 by arranging several digits in such a way that each position in the number means something different.

For example the number "245" means really "2 times 100 + 4 times 10 + 5 times 1". The further left a digit's position in the number, the higher a value it contributes to the number. Every time you go one step left in the number, you multiply the digit there with 1, 10, 100, 1000 and so on and add that value to what you already have. The values you have to multiply the digit with can be obtained if you start out with 1 on the far right, then multiply with 10 every time you go left. So you get to multiply with 1, then with 1×10=10, then with 10×10=100, then 10×100=1000 and so on. This is our "base 10" number system.

Now the binary system is a "base 2"-system. A binary number might looks like this: 1011

This is not "one thousand and eleven", as it would be in the decimal system. In the decimal system we multiplied the digits at the various positions with a 1, 10, 10×10=100, 10×100=1000 and so on, but this time, in the binary system (also known as "dual system"), we multiply with 1, 2, 2×2=4, 2×4=8, 2×8=16 and so on, starting at the right and doing with a "2" what we did with a "10" before. So the value of the binary number above is "1 times 1 + 1 times 2 + 0 times 4 + 1 times 8", giving us the number "11" in our decimal system.

Computers use bits to calculate everything, because they can be modeled as "current on" or "current off" easily. With a packet of 4 bits (called a "nibble") you can describe the numbers from 0 to 15. A packet of 8 bits is called a "byte", and these are what the file contains. Each byte can describe a number from 0 to 255. So, a file is a list (or "array") of "bytes", with each byte at a certain index or "offset". These bytes are what frhed displays in the middle of the screen, but in yet another different number system: the "hexadecimal" system, which is a system with the base 16. So there are 16 digits, the normal ones from 0 to 9, and additionally the letters "a" through "f" with following values:

a = 10
b = 11
c = 12
d = 13
e = 14
f = 15

So the "hex" (short for "hexadecimal") number "f14a" means: "10 times 1 + 4 times 16 + 1 time 16×16(=256) + 15 times 16×256(=4096)", giving a value of 61770 in decimal. The point of using hex numbers is that values of up to 255 (maximum value of a byte) can be described by employing up to 2 hexadecimal digits, which gives a neat and ordered output on hex editors such as frhed. On the right side of the frhed window there are "normal" letters, punctuation symbols and so on. This is because each byte can be seen as an index to a symbol in a "character set". This means that the computer stores for example the letter "a" as a number with the value 97 ("61" in hex notation), the comma "," as a number 44 and so on. So if you write a text on the computer and save it to disk, then the file will contain a number of bytes, each byte signifying a letter, other symbol, or a special code number for "End of line" or "Tabulator" and such. But there are several character sets in use (for example Windows uses either the ANSI or the OEM character set): In one character set the number 97 might mean the letter "a", but in another it could be used for the apostrophe.

File operations

The following File Operations are supported:

  • Creating new files
  • Opening files
  • Opening files partially
  • Saving files
  • Saving selections
  • Reverting to the saved file
  • Deleting the current file
  • Exporting the file as a hexdump text
  • Importing a hexdump file

Creating new files

If frhed is started without parameters or with a nonexistant filename as parameter, then it starts with a new, empty file that is untitled. If a file is already loaded, then choose File/New from the menu. If a file is already loaded and it has been changed, then you will be prompted if you want to save the changes.

Opening files

If frhed is started with command line parameters, then the whole string following the program name is considered to be the name of the file to open.

Examples:

frhed filename.dat opens the file called "filename.dat"
frhed a file-name.dat opens the file called "a file-name.dat"

Note: If you put a link to frhed.exe into the "SendTo"-Folder in your Windows main directory then you can open any file from the Explorer with frhed by choosing frhed from the "Send To"-item in the file's context menu.

You can also choose File/Open from the menu. If a file is already loaded and it has been changed, then you will be prompted if you want to lose the changes, save the changes or cancel the request.

A file can also be dragged (for instance from the Windows Explorer window) into frhed's window to open it. If several files were dragged, then you must choose which one to open.

You may also select a file from the MRU list to open.

If the file access permissions don't allow writing or open-read-only mode is on then read-only mode will be engaged.

Opening files partially

If you want to edit a very large file and it's not necessary to load all of it into memory, choose the File/Open partially command. First you will have to choose the file you want to open in a File-Open-dialog. Then you must enter the offset from which the file will be loaded and how many bytes to load, or you can check the "Open partially from end of file"-radiobutton if you want to load data from the end of the file. Also you may have frhed display the offsets relative to the start of the file instead of relative to the start of the loaded block. Note, however, that so far the rest of the program uses offsets relative to start of the loaded block.

frhed will now be in partially-opened-mode, which means that whenever you choose the File/Save command, the data will be written back to the position they were loaded from. This mode is indicated by a letter "P" in the window title after the file name.

If the file has been changed, then you will be prompted if you want to lose the changes, save the changes or cancel the request.

You can also resize the block of data you loaded and the corresponding section of the file will also be resized when you save the file.

If the file access permissions don't allow writing or open-read-only mode is on then read-only mode will be engaged.

Saving files

To save the current file choose File/Save from the menu. If the file has never been saved before, then File/Save as will be automatically invoked.

File/Save as saves the file with a different name.

Saving selections

To save the current selection choose File/Save selection as from the menu. If the is no selection then all of the file will be saved.

Reverting to the saved file

When you have made changes to a file, want to change it back and cannot remember the exact bytes in the file then the File/Revert command will do it for you. If the file has never been saved then this command will revert to an untitled empty file.

Exporting the file as a hexdump text

The File/Export as hexdump command is used if you need the hexdump of the file in plain text form, for example if you need to insert it in a word processor document. You can either put the hexdump into the clipboard or into a file, which you will have to choose. The start line of the hexdump has to be specified (in hexadecimal notation without prefix), which you can find on the left side of the window, for example 23c or 134. You must also specify the end line. The values in the dialog will be initialized so that the whole file will be dumped, if there is nothing selected, or they will be such that the current selection will be dumped.

Also you have the option of just outputting the hex bytes on a single line - just uncheck the "Export like frhed display" check box.

Also you have the option of outputting an RTF hexdump of the lines. The RTF is output by frhed as follows:

  • The file that was hexdumped in the document properties as the Title.
  • A Times New Roman 20pt paragraph containing the file name of the file that was hexdumped.
  • Each line of hexdump, with the selection or current byte highlighted as per the display colours.
  • Bookmarks are both highlighted and put into the document. The bookmark name is that same as what you gave frhed, unless you didn't give a name, in which case the name will be the offset of the bookmark in decimal. The document bookmarks will be in the same place as the caret.
  • The separators are put in a single group together and are simple lines as per the display settings.
  • Limitations: the RTF output is not perfect in these ways:
    For the chars Microsoft Word will convert some into underscores (generally the control characters up to 0x20). This is only apparent in OEM mode since in ANSI mode frhed translates these chars into periods (.).
    Microsoft Word will alter the bookmark names by replacing non-alphanumerics with underscores, place "BM" in front of bookmarks that begin with a number and hide bookmarks beginning with underscores (can still view them with "Hidden bookmarks" checked).
    If your display is running at anything less than 24 bit colour you may not get the colours you expect for the caret. This is because at lower bit levels the InvertRect function (which frhed uses to invert the caret in the window) works differently than just inverting the bits of colours as it does at 24-bit & as frhed does in generating the inverted colour for RTF export. If someone finds out how InvertRect works at lower bit levels please let me (Pabs) know.
    The bookmarks border takes up space (at least on Word2000) which means that subsequent characters on the same line will be displaced to the right 2 pixels. This is noticable when there is a selection covering two adjacent lines, one with & one without a bookmark. There is little that can be done about this except try a different word processor.
    The separators will not always be sized or positioned exactly. There is not much that can be done about this since it is hard to know the exact size in twips the word processor will display the text at. An alternative to this might be to use tables, which might not work and might also solve the bookmark problem.
    At some fonts & sizes (esp. OEM & ANSI 9, ANSI 10) frhed cannot determine the correct font name the system chose & this will cause Word to use the wrong font, often this will be Times New Roman. This may or may not be true depending on whether or not the font mapper for your system uses one font for all sizes and the fonts you have installed. Unfortunately Word ignores the other font hints that can be determined; character set, variable/monospaced and font family.

Deleting the current file

USE WITH CAUTION

The File/Delete File command deletes the file currently opened in the editor and closes it. This requires a double confirmation because it could be disastrous.

When ever you edit files using any hex editor you should always have a backup anyway.

Importing a hexdump file

The File/Import from hexdump command imports hexdump data from the clipboard or from a file.

The data you choose to import can be in one of two formats. One is a text version of the display & the other is all the hex digits on one line.

The frhed will try to determine the type by looking for characters that are not whitespace or hexdigits and you will be asked to verify the frhed's findings.

If you decide it is in the frhed display format then you will be asked if you want the display settings found in the file to replace current ones. If so, frhed can determine the offset length, the number of bytes per line, if the offset length should be automatically adjusted and possibly if the data is ANSI/OEM from its contents. If any of these cannot be determined their settings will remain the same.

frhed will also notify you if the character and hex data do not agree with each other or the offsets are invalid and you may choose to ignore any more of these notices.

If illegal characters are found in the hexdata section or the data ends unexpectedly the operation will not be able to continue and you will have the option of keeping the data that has already been found or cancelling the operation.

Also if the first offset found is not zero you can insert that amount of null bytes at the start of the file and if you choose not to and you chose to alter relevant display settings then frhed will display offsets relative to the first offset found.

If you decide it is not in the frhed display format then frhed will go through the file filtering out whitespace and adding each hex digit found as a nibble (If odd no's of hex digits are present then '0' will be the last nibble). If other characters are found frhed will complain and give you the option of ignoring the any other illegal characters that are found.

If the file access permissions don't allow writing or open-read-only mode is on then read-only mode will be engaged.

If the current file has been changed, then you will be prompted if you want to lose the changes, save the changes or cancel the request.

BUGS:

  • If the file was output using an offset length that is smaller than is required the parser will determine that the character data does not match the hex data. If the output data is edited to the same as with an offset length of 2 the same thing happens. This occurs because of a bug in File/Export as Hexdump, causing it to remove characters from the end of each line that has larger offset lengths than that of the first line. This bug was fixed in v1.0.156.Pabs. If you know the file was output using a small offset length and are sure that you have not edited it press Yes to ignore any further mismatches, no to ignore that one or cancel to cancel the Import operation. Ignoring the mismatches will have no effect.

The main window

The window's title shows the name of the currently loaded file in square brackets, with an asterisk on it's right side if the file has been changed. If a file is loaded, you will find something like this in the window's client area:

000000  23 42 34 53 56 45 76 45  #B4SVEvE
000008  64 56 0a bc 0a 35 c0 08  dV...5..
000010  06 be 09 07 bb 07 00 00  ........
000018  00 00 00 00 00 00 07 __  ....... 

On the far left are the offsets of the first byte in the line in hexadecimal notation. You can choose the offset length or make it automatically adjusted to the length of the largest offset with the Options/View Settings command or right-click the offsets to increment their lengths or left-click them to decrement their lengths (when auto-adjust is off). Then come the actual byte values, with vertical separating lines after every fourth byte in the line. You can choose the number of bytes to display per line with Options/View Settings, and also, if it should be automatically adjusted to the window width. To the far right are the character representations of these values, with dotted lines on the left and right of this section. You can choose the character set for these representations as well as the font size with the Options/Character set command. As you can see, the first byte of the second line of the sample hexdump has the hexadecimal value 64 and the corresponding character is a lower case "d". The position after the last byte of the file is marked with a double underscore "__" in the byte section.

Blinking in the window somewhere is a rectangular caret, if the window is enabled. If it is in the byte section, then it stands on either of the two nibbles (half-bytes). If it is in the character section, then it stands on one character corresponding with one byte. In either case the corresponding other representation is shown inverted in the other section.

On the bottom of the window is the status bar, which displays additional information. In the first section of the status bar you can find the current offset of the byte which the caret is marking in hexadecimal and decimal notation. After that come the bits of the current byte. Also in this section, on the right side, is something like this:

Unsigned: B:35,W:16931,L:1395933731

These are the values of the byte (B:), the word (W:) and the longword (L:) value starting at this offset in decimal notation. These are computed depending on the setting for signed or unsigned values (Choose with the Options/View Settings command) and on the binary mode setting, either Little-endian (Intel format) or Big-endian (Motorola format). Choose with the Options/Binary mode command.

In the second section of the status bar are indicators for:

  • character set: "ANSI" or "OEM".
  • entering mode: "OVR" for overwriting or "INS" for inserting on keyboard input. If frhed is in read-only mode, then there will be a "READ" here.
  • binary mode: "L" for Little-endian or "B" for Big-endian.

In the third section the file size in bytes is displayed.

The status bar can also be used to toggle various parameters and act as a shortcut to certain commands.

The status bar takes into account the same factors when considering if the selected command should be carried out as the menus do when disabling commands. If there are 2 alternatives in one place and one is disabled then the other one will be used.

A single left click in the third section is a shortcut to Edit/Append (right is Edit/File properties) - Note that this may change in the future.

Clicking in the middle section will toggle the parameter clicked on - ANSI ↔ OEM, READ → INS → OVR → READ… (right-click goes back other way) , L ↔ B.

Left or right clicking in the first section can result in many different things depending on where you click and state of the file opened.

When part of the file is selected you may deselect (cursor goes to the first nibble of the first byte of the selection or last of last for a right click) by clicking in the space on the right or the word "Selected:" or alter the selection using Edit/Select block by clicking in the offsets region.

If the file is empty clicking the first part will bring up Edit/Append.

If the file is not empty and the cursor is on the end byte then Edit/Goto will be activated on a left click or the cursor goes to the previous byte on a right click.

Otherwise (when not on the end byte, selected or empty) the click behaviour is complex and summarised in the colour-coded table below. The right action is omitted where it is the same as the left one.

If you have suggestions as to what else the status bar could/should (not) do just drop me (Pabs) a line.

The window also has a vertical and a horizontal scrollbar, which can be used to scroll the view. They don't change the caret position.

Moving in the file

You can move the caret with the arrow keys. Press TAB to toggle the editing mode between byte and character editing. When there is a selection TAB brings the selection into the middle of the display.

You can also jump to any position by choosing the Edit/Go To command and entering an offset, like "125", which will set the caret position to offset 125 decimal. For relative jumps in the file, for example if you want to go to an offset 150 bytes after the current one, you have to start with "+" as the first character and follow this with either a decimal number or a hex number starting with "x". So to jump 150 bytes forward you have to enter "+150" or "+x96" in hex notation. For backward relative jumps start with "-" (minus) as the first character.

To jump to the start of the line press Home or End for the end of the line. To jump directly to the end of the file press Ctrl+End, for the beginning Ctrl+Home.

frhed can find text and binary values in a file. Choose Edit/Find to do this. In the dialog you can choose if the case of the text parts of the find string are to be ignored and the find direction. To encode binary values, frhed uses a special syntax. Example:

Here is the findstring<bd:0>FloatValue<fl:-14.56>

This will cause frhed to search for a string "Here is the findstring." followed by a zero-byte, then the text "FloatValue" and a 4-byte float with the value -14.56.

Note that coded binary values will also be transformed by the "match case" option of the Find command, so if you search for a byte with the hex value of 0x41 (this is the upper-case letter "A") with the match-case option disabled, then frhed will also find bytes with the hex value 0x61 (the lower-case letter "a"). Therefore you should always search for discrete binary values with case-matching enabled or you might not get correct results. Please see Using the special syntax for further information.

The Edit/Find next and Edit/Find previous commands find the next or previous occurance of the findstring last entered in the Find-dialog, with the match-case option enabled or disabled as chosen in it. If there was no previous findstring the both Find next & Find previous commands will bring up the Edit/Find dialog.

By positioning the mouse cursor on a byte or a character and pressing the left mouse button you can also set the caret position.

You can also move in the file by choosing one of the bookmarks in the "Bookmarks" menu section. The caret will be set to that offset immediately.

Using the mouse

frhed implements OLE drag-drop to allow you to move data in a file, between frhed instances and even between frhed and other applications. It follows similar standards to most OLE apps, but when dropping and there is more than one format you must choose which format(s) to insert and you need to specify which format(s) to transfer before you can drag data to other apps. To specify the formats to use and how to use them for OLE Drag-drop choose Options/OLE Drag-drop. Currently frhed can output "BinaryData" (used for transfers between frhed instances), both types of hexdump as CF_TEXT, the special syntax as CF_TEXT and Rich Text Format for pasting properly formatted & coloured hexdumps into wordprocessors

When moving or copying data around in the file you don't have to choose which format as frhed just does what you wanted.

When you drag data over frhed a flashing cursor shows where the data will be inserted.

Below is a summary of the keys that can be used to control the drag operation and their effects.

ButtonEffect
ESCCancel dragging operation
Left mouseMove data
Right/middle mouseChoose either Move or Copy from a menu/dialog on the drop
CtrlCopy data
ShiftOverwrite data (can be combined with both Move and Copy)

Note that the Overwrite mode of the file is ignored when dropping. To overwrite existing data with that being dropped use the shift key.

Using the keyboard

For overwriting data the overwriting entering mode must be enabled, that means, the status bar must show "OVR" somewhere. Toggle the entering mode with the Insert-key on the keyboard or choosing Edit/Toggle entering mode. You can overwrite the data in the bytes section by moving the cursor to the appropriate position and pressing any of the hexadecimal digits 0-9 and a-f on the keyboard. The nibble the caret is marking currently will be set to that value and the caret will then go on to the next nibble. If the caret is in the bytes section, pressing any alphanumeric key will set the current byte (that wich the caret is marking) to that value, and the caret will go on to the next byte.

For inserting data the status bar must show "INS" somewhere. Entering data now will insert a byte at the current position.

If there is a selection then you can replace it with a single character.

The Edit/Cut command

This command is used to remove bytes from the file. These bytes can either be copied to the clipboard (as text with special syntax for non-character bytes) or just deleted. In the dialog you can choose the start offset where the cutting begins (the byte at that offset is the first to be cut), how many bytes to cut (specify either the offset of the last byte to cut or the number of bytes including the first byte to be cut, and check the appropriate radio button). If the data should be cut to the clipboard, check the checkbox on the bottom of the dialog box.

If the command is chosen without a selection, then the values in the dialog will be set so that the byte at the current offset will be cut.

The Edit/Copy command

This command is used to copy bytes from the file to the clipboard. It is similar to the Edit/Cut command, only the data will only be copied, not removed.

The Edit/Paste command

This either inserts or overwrites data from the clipboard starting at the current offset. Choose a radio button for the writing mode wished for. You can also choose how many times the current data is written and how many bytes to skip between each write of the data.

If "Paste coded binary values as text" is unchecked, all data in the clipboard is considered to contain special syntax, so <ld:150> will not be written as the text "<ld:150>" but as a longword value of 150. Check the checkbox if you want to paste as text. The binary mode (Little or Big-endian) and character set (ANSI or OEM) is used as set in the options.

You can also choose which clipboard data you want to paste.

If one of the text formats (CF_DSPTEXT, CF_OEMTEXT, CF_TEXT and CF_UNICODETEXT) is chosen trailing null characters are not pasted. Also pasting CF_UNICODETEXT doesn't work unless "Paste coded binary values as text" is checked. This is because the translation routine only works with single byte text. Don't count on it but I (Pabs) may implement a Unicode version in the future.

If any other format is used the full binary data is pasted. As a consequence of the way Windows works, however, there may be some null/garbage bytes at the end of the pasted data. This is fine (besides being wasteful) but Windows does not provide a way to look up the requested size of the data but only the allocated size.

The Options/Binary mode command

What kind of byte order to use: Either Little-endian, as used in Intel-computers or Big-endian, used for example in computers with a Motorola processor. The difference between these modes is thus: Consider a longword, which is 4 bytes (named A to D) long.

Longword in processor register = ABCD.

The least significant bit, which has the value 1, is found in byte "D". In a processor register the longword is ordered in this way. If this value is written to a position in memory, then by Intel convention the longword is written in this way, with the growing offsets to the right:

Longword in memory = DCBA.

That is, the bytes making up the longword are written in reverse order as they are found in a processor register.

This means that a file written by a Big-endian machine will contain wrong longword and word values if read by a Little-endian machine without conversion.

Using the special syntax

Byte, word, longword and floating point data can be encoded as follows:

<{Size}{Numeral system}:{Value}>

Size:

b for byte-size (8 bit) value.
w for word-size (16 bit) value.
l for longword-size (32 bit) value.

Numeral System:

d for decimal system.
h for hexadecimal system.

For floating point values set Size and Numeral system to:

fl is for float values.
do is for double values.

Value:

Enter the value of the datum after the colon.

Examples:

<bd:-15> means byte-size datum with decimal value -15.
<wh:23f0> means word-size datum with hexadecimal value 23f0.
<ld:1234567890> means longword-size datum with decimal value 1234567890.
<fl:2.5> means float-size datum with value 2.5.
<do:1.8e7> means double-size datum with value 1.8×10^7.

Note: If you want to find or paste a valid special syntax string like <bd:0> as text, then prefix a \ to <. This will turn < into a normal character. Copying a < in the file will insert the \ automatically.

Backslashes ("\") in the file are copied and encoded as double backslashes ("\\").

Data ranges and sizes:

Integers:

Byte:

signed from -128 to 127
unsigned from 0 to 255

Word (2 bytes):

signed from -32,768 to 32,767
unsigned from 0 to 65,535

Longword (4 bytes):

signed from -2,147,483,648 to 2,147,483,647
unsigned from 0 to 4,294,967,295

Floating point types:

Float (4 bytes):

range 3.4E +/- 38 (7 digits)

Double (8 bytes):

range 1.7E +/- 308 (15 digits)

Shortcuts to frhed

This is a link manager specific to frhed. All links to frhed created by this have the filename "frhed.lnk". You can add new links by specifying paths to put them in, move existing links to new paths, delete links one at a time, reload the list of links from the registry, find links to current & old versions in the file system and create links in any of four predefined places:

  1. The Desktop
  2. Start menu
  3. Programs menu
  4. Send To menu

The paths of these are read from the registry and should always point to the correct path. The paths of the links (without names) are stored in the registry under HKCU\Software\frhed\<current version>\links. All values in this key that contain strings are loaded into the list of links after being verified and or updated.

Note that:

  • The link manager tries to keep the registry & file system up to date as much as possible and if you delete a link from explorer & load the link manager the all items in the registry will be recreated.
  • Duplicates are not allowed and if you delete one of the values from the registry then frhed will not be able to delete the corresponding link from the file system.
  • You cannot create links on read-only devices as the browsing mechanism detects volumes where files cannot be created. These include places that do not have a path in the file system, like places such as Control Panel etc., places on read-only media and places on volumes that are full.
  • Renaming a link will allow you to delete it from the registry but not the file system.
  • If there are links to removable media you may see DVD/CD/floppy activity when frhed is searching for links.
  • Note that to have links to more than one version of frhed (that has the shortcut manager) you will have to create one link, rename it by appending a version no and then create the other link and rename it in the same way. This will ensure that if you accidentally add a link to frhed to a folder then links to any old versions will not be overwritten. These links will not be safe from the find n' fix function - to make them safe rename the exe by appending a version number and create a link to the exe that is not named frhed.lnk (to prevent it being overwritten).
  • Update & find n' fix both search recursively from a user specified directory for links to frhed - update looks for links to the filename of the running version of frhed while find n' fix looks for links to any frhed.exe that is in the search tree. Duplicate links in the same directory will be removed and all links found will be renamed to frhed.lnk.

Frequently Asked Questions

Q: How do I search for hex values?

A: You have to describe the values using a special syntax (as described in "Using the special syntax" in the help file). For example the byte value 0xf7 has to be written as (you must write the angled brackets too!) 

<bh:f7>
where the b means byte-sized value (otherwise use a "w" for word-size or "l" for longword-size) and the h means hexadecimal notation (otherwise use "d" for decimal notation). This notation can be combined with text after it or before it, or several encoded values after another etc.

Q: frhed opens all files from Explorer supenly. How do I remove this behavior?

A: Uncheck the following: Registry/In all context menu, Registry/Is default in unknown context & Registry/In unknown context menu. Or edit the registry - Delete HKCR\*\shell\Open in frhed\, HKCR\Unknown\shell\Open in frhed\ & the "(Default)" values under their parent keys.

Q: What do the B:??, W:??, L:?? in the status bar mean?

A: These show you the byte (B), word (W) and longword (L) value beginning at the current cursor position, interpreted as either signed or unsigned, which is indicated in front of the "B:".

Q: I know that two files I am comparing with the Edit/Compare from current offset command are equal, why does frhed show me lots of differences?

A: As the command's name indicates, the file you choose from the following dialogue is compared to the currently loaded file beginning from the CURRENT CURSOR POSITION. So if you want to know if the two files match completely beginning to end, you must put the blinking cursor in the very first position of the file, that is, at index zero. Just press Ctrl+Pos1 on the keyboard to go there directly.

Q: My Wheel Mouse does not work correctly with frhed. Are wheel mice supported?

A: I also use a Logitech wheel mouse, which works fine with frhed. It is possible that the standard drivers installed by Windows for wheel mice do things slightly differently than the "real" Logitech drivers. I suggest you install the original drivers supplied by your mouse manufacturer. If you have any success with this, please tell me about it, I don't know if this really works.

This help file uses CSS or "Cascading Style Sheets" to change the display characteristics of text, lists, images, tables and pretty much everything else. NOTE: Stylesheets may only work with IE4 and above.

All topics in this help file (even this one) reference three CSS files in the following order:

  • before.css is located in the same directory as this help file (frhed.chm).

  • frhed.css is located inside this help file and contains the default styles for this help file.
  • after.css is located in the same directory as this help file.
    Use this stylesheet if you want to override both the browser defaults and frhed's defaults.

NOTE: before.css and after.css are accessed using JavaScript, so you must turn it on before you can use these style sheets. The alternative to using JavaScript would have been to put them in the root directory of the disk where this help file is located and accessed them using static links to file:///before.css and file:///after.css. The JavaScript method, however, is much more flexible and prevents the need for write access to the root directory, which would probably not be granted on locked down machines, such as in university computer labs and work machines. If you prefer to use static links you can do so by installing the help file source, replacing all the JavaScript stuff with static links, regenerating the individual topics from the big-flat-file using pydocgen.py and recompiling using HTML Help Workshop from Microsoft.
e.g. This:

<script language="JavaScript">document.createStyleSheet(externlink("after.css"));</script>

is converted to:

<link rel="stylesheet" type="text/css" href="file:///<your static path>/after.css">

  • WWW Consortium's CSS2 Specification - The official specification.
  • Microsoft's "Cascading Style Sheets in Internet Explorer 4.0" - what is supported by HTMLHelp

frhed.css is available to study and use in before.css and after.css.

Tips:

  • You can import other stylesheets using @import "mystylesheet.css";. The semicolon(;) is important and the import must be before the definition of any styles.
  • You set styles using the following syntax:
    Note the semicolon after each value.
    There is more to the syntax than what is listed here - please visit the above two references. 
    [<some-tag>][.<some-class>][:<some-pseudo-class>] {
	(<some-attribute>: <some-value>;)...
}...
    VariableExplanation
    <some-tag>The name of a HTML tag e.g. body, a, ul.
    <some-class>The value of the "class" attribute in a tag in the document eg. .SomeClass { color: red; } (or p.SomeClass) in CSS then <p class="SomeClass">Some text</p> in HTML.
